# コーディング規約

> プロジェクト: **godufo-game（宇宙大戦シューティングゲーム）**  
> 作成者: Takumi Harada  
> 作成日: 2026/04/06  

---

## 1. 概要

本ドキュメントは godufo-game における JavaScript のコーディング規約を定めます。  
全ファイルはこの規約に沿って実装・レビューすることを原則とします。

---

## 2. フォルダ構成

```
src/
├── core/
│   ├── game-core.js            # ゲーム全体の進行制御・メインループ
│   └── game-player.js          # プレイヤーの状態・描画
├── data/
│   ├── code-definitions.js     # コード定義（敵タイプ・アイテム種別等）
│   └── enemy-data.js           # 敵キャラクターのデータ定義
├── managers/
│   ├── collision-manager.js    # 当たり判定処理
│   ├── enemy-manager.js        # 敵の生成・移動・描画
│   ├── input-manager.js        # キーボード・タッチ入力の受付
│   ├── projectile-manager.js   # 弾（自弾・敵弾）の管理
│   ├── render-manager.js       # Canvas への描画処理
│   ├── special-manager.js      # 必殺技エフェクト処理
│   ├── state-manager.js        # ゲーム状態（スコア・フラグ等）管理
│   └── ui-manager.js           # DOM 上の UI 更新（スコア表示・シールドバー等）
├── ui/
│   ├── components.js           # 再利用可能な DOM 生成関数
│   ├── header.js               # ヘッダー HTML 生成
│   └── footer.js               # フッター HTML 生成
└── utils/
    ├── constants.js            # ゲーム全体の定数（GAME_CONFIG・INPUT_KEYS 等）
    ├── validation.js           # 入力バリデーション関数
    └── xss.js                  # XSS サニタイズ関数
```

### 各ディレクトリの責務

| ディレクトリ | 責務 |
|---|---|
| `core/` | ゲームの核。メインループ・プレイヤー制御 |
| `data/` | 変更しないデータ定義のみを置く。ロジックを含めない |
| `managers/` | 単一の機能に特化したクラス（当たり判定・描画・入力等） |
| `ui/` | DOM 生成・更新のみ。ゲームロジックを含めない |
| `utils/` | プロジェクト横断で使える純粋関数・定数 |

## 3. 命名規則

### 3-1. ファイル名

- **kebab-case** を使用する

```
game-core.js        ✅
gameCore.js         ❌
```

### 3-2. クラス名

- **PascalCase** を使用する

```js
export class Game { }           ✅
export class EnemyManager { }   ✅
export class enemyManager { }   ❌
```

### 3-3. 関数・メソッド・変数名

- **camelCase** を使用する

```js
getUfoStyle() { }   ✅
get_ufo_style() { } ❌
```

### 3-4. 定数名

- **UPPER_SNAKE_CASE** を使用する
- ゲーム設定値・入力キーは `utils/constants.js` に集約する

```js
export const GAME_CONFIG = { CANVAS_WIDTH: 480 };  ✅
const canvasWidth = 480;                            ❌（マジックナンバー禁止）
```

## 5. コメント規約

### 5-1. ファイルヘッダー

全ファイルの先頭に JSDoc ブロックコメントを記述する。

```js
/**
 * <ファイルの役割を一行で>
 * @author Takumi Harada
 * @date YYYY/MM/DD
 */
```

### 5-2. メソッドコメント

メソッドの役割を一行で説明するコメントをメソッドの直前に付ける。

```js
// ゲームが未開始またはゲームオーバー後は更新処理をスキップする
update() {
  ...
}
```

### 5-3. if 文コメント

**すべての `if` 文の直前行**に、何を判定しているかを日本語で説明するコメントを付ける。

```js
// ゲームが開始前またはゲームオーバー後は発射を無視する
if (!this.isStarted || this.isGameOver) return;
```

### 5-4. 複雑なロジックのコメント

自明でないロジック（減衰式・当たり判定式等）には理由を補足する。

```js
// シェイクは毎フレーム 0.92 倍で減衰する
this.shake *= 0.92;

// ホーミング弾かつ生きている敵がいる時は追尾計算を行う
```

### 5-5. クラス内の構造コメント

クラスが長くなる場合はセクションを区切るコメントを使う。

```js
// ゲーム進行フラグ
this.isStarted = this.isGameOver = false;

// スコア・フレームカウンタ・シェイク量
this.score = this.frame = this.shake = 0;
```

## 6. モジュール設計

### 6-1. ES Modules 使用

`import` / `export` を使用する。`require()` は使用しない。

```js
import { Player } from "./game-player.js";
export class Game { }
```

### 6-2. 名前付きエクスポートを原則とする

```js
export class EnemyManager { }          ✅
export default class EnemyManager { }  ❌
```

### 6-3. import パス

拡張子 `.js` を明示する。

```js
import { GAME_CONFIG } from "../utils/constants.js";  ✅
import { GAME_CONFIG } from "../utils/constants";     ❌
```

## 7. クラス設計

### 7-1. 単一責任の原則

1 クラス = 1 つの責務。ゲームロジックと描画を同一クラスに混在させない。

| クラス | 責務 |
|---|---|
| `Game` | メインループ・Manager の指揮 |
| `CollisionManager` | 当たり判定のみ |
| `RenderManager` | Canvas 描画のみ |
| `UIManager` | DOM 上の表示更新のみ |
| `InputManager` | 入力受付のみ |

### 7-2. Manager パターン

各 Manager は `Game` から参照される。Manager 同士が直接参照し合うことは避ける。

```js
// game-core.js 内
this.collision = new CollisionManager();
this.render    = new RenderManager();
```

## 8. セキュリティ

### 8-1. XSS 対策

ユーザー入力や外部データを DOM に反映する際は必ず `xss.js` のサニタイズ関数を通す。  
`innerHTML` への直接代入は禁止。

```js
import { sanitize } from '../utils/xss.js';

element.textContent = sanitize(value);   ✅
element.innerHTML   = value;             ❌
```

## 9. パフォーマンス

### 9-1. requestAnimationFrame

メインループは `setInterval` ではなく `requestAnimationFrame` を使用する。

### 9-2. 死亡判定の早期スキップ

死亡済みオブジェクトの更新・描画は先頭で早期リターンする。

```js
// 既に死亡した敵は更新処理をスキップする
if (enemy.isDead) return;
```

### 9-3. マジックナンバー禁止

Canvas サイズ・ゲーム速度・ダメージ値などの数値はすべて `utils/constants.js` の定数で管理する。

## 10. Git 運用

### 10-1. コミットメッセージ

プレフィックスを付けて種別を明示する。

| プレフィックス | 用途 |
|---|---|
| `feat:` | 新機能追加 |
| `fix:` | バグ修正 |
| `refactor:` | リファクタリング（動作変更なし） |
| `docs:` | ドキュメント更新 |
| `chore:` | 設定・ツール類の変更 |

```
feat: ホーミング弾アイテムを追加
fix: 必殺技エネルギーが溜まらない問題を修正
refactor: ifブロックに判定内容コメントを追加
docs: コーディング規約を追加
```

### 10-2. ブランチ

| ブランチ | 用途 |
|---|---|
| `main` | 本番リリース用 |
| `feature/xxx` | 機能開発 |
| `fix/xxx` | バグ修正 |
